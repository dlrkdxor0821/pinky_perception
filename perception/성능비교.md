# 성능 비교: 온디바이스(엣지) vs AI 서버

같은 객체탐지 작업을 **(1) 라즈베리파이에서 직접(NCNN)** vs **(2) 로컬 AI 서버로 보내서(GPU)**
했을 때의 성능을 비교한다. 클래스: `person`, `mobile_robot`.

| 항목 | 엣지(온디바이스) | AI 서버 |
|------|------------------|---------|
| 추론 위치 | 라즈베리파이 Pi 4 (Cortex-A72) CPU | 로컬 PC (GPU) |
| 모델 | NCNN 경량, imgsz 320 | full 모델 (PyTorch) |
| 네트워크 | 없음 | UDP/HTTP 왕복 |

---

## 1. 온디바이스 (엣지) — ✅ 측정 완료

**측정 조건**: Pi 4 (Cortex-A72, 4코어) / NCNN raw / imgsz 320 / conf 0.6 / threads 4 /
CSI 카메라 실시간 / **300 프레임** 연속. (`benchmark/results/edge.csv`)

| 지표 | 값 |
|------|-----|
| **FPS** | **5.31** |
| 지연 평균 | **165 ms** |
| 지연 p50 | 150 ms |
| 지연 p95 | 242 ms |
| 지연 p99 | 286 ms |
| **CPU 사용률** | **84.0%** (min 66 / max 90) |
| **RAM 사용량** | 1687 MB (min 1673 / max 1690) |
| **SoC 온도** | **60.4 ℃** (max 65.2 ℃) |
| **Throttling** | `0x0` (스로틀 없음) |

> 자원 지표 출처: `benchmark/results/edge.csv` (300프레임 집계). 엣지는 4코어를
> 거의 풀로(CPU ~84%) 쓰지만, idle(49℃)에서 시작해 이번 측정 중엔 60℃에 머물러
> **스로틀이 걸리지 않은 상태** → 5.3 FPS 유지.

**참고/주의**
- 위 5.3 FPS / 60℃는 Pi가 식은 상태에서 시작한 **스로틀 없는** 측정치. 단, 더 오래
  지속 부하를 주면 발열로 76℃ + **스로틀(`throttled=0xe0000`)** 에 도달해 ~4.3 FPS /
  지연 ~217ms로 떨어짐 (이전 측정). 전원(5V 3A↑)·쿨링 개선 시 스로틀 회피 여지.
- fp16/int8은 A72에서 효과 없음(상세: `ONDEVICE_EFFORTS.md`).

## 2. AI 서버 경로 — ✅ 측정 완료

서버 PC(GPU)에 full `best.pt` 띄우고, Pi에서 카메라 프레임 전송, **300 프레임** 측정.
(`benchmark/results/server_udp.csv`, `server_http.csv`)

**FPS·지연은 네트워크 상태에 크게 좌우되므로 두 조건 모두 기록**한다. 실사용 환경은
보통 공용/혼잡 WiFi이므로 **(a) 망 혼잡 조건을 대표값**으로 삼고, 전용 LAN(학원)에서의
값은 best-case 참고로 둔다.

### (a) 망 혼잡 — 실사용 환경 (공용 WiFi) ⭐ 대표값

| 지표 | UDP | HTTP |
|------|-----|------|
| **FPS** | **13.8** | **11.4** |
| end-to-end 지연 평균 | **48.4 ms** | **62.3 ms** |
| 지연 p50 / p95 / p99 | 32.0 / 129.3 / 230.1 ms | 56.5 / 95.3 / 202.9 ms |
| 서버 추론 시간(GPU) | 7.4 ms | 5.0 ms |
| 네트워크 왕복(= e2e − 추론) | 41.0 ms | 57.3 ms |
| 프레임 유실률 | 0 / 300 | 0 / 300 |
| 극심한 혼잡 시 UDP | **프레임 거의 전멸** (datagram 손실 → 재조립 실패) | — (TCP라 재전송으로 버팀) |

### (b) 망 정상 — 전용 LAN(학원) best-case (참고)

| 지표 | UDP | HTTP |
|------|-----|------|
| **FPS** | 27.3 | 18.7 |
| end-to-end 지연 평균 | 29.7 ms | 44.5 ms |
| 지연 p95 / p99 | 61.1 / 143.1 ms | 78.2 / 155.5 ms |
| 네트워크 왕복 | 22.3 ms | 39.1 ms |
| 프레임 유실률 | 0 / 300 | 0 / 300 |

> **UDP가 HTTP보다 빠름** (오버헤드 적음): 혼잡망 e2e 48.4 vs 62.3ms.
> 핵심: **서버 지연의 대부분은 네트워크**(혼잡 41~57ms / 정상 22~39ms)이고 GPU 추론
> 자체는 5~7ms로 매우 빠름. **FPS는 네트워크 의존**이라 망 상태에 따라 14↔27처럼
> 2배 차이날 수 있음. UDP는 손실에 취약 — 혼잡 시 멀티-datagram 프레임이 통째로 깨져
> 유실↑ (HTTP는 TCP라 느려도 버팀).

측정 명령:
```bash
# 서버 PC
python3 perception/scripts/run_server.py --weights <full>.pt --device cuda
# Pi
perception/.venv/bin/python perception/client/detect_client.py --transport udp  --host <서버IP> --frames 300
perception/.venv/bin/python perception/client/detect_client.py --transport http --host <서버IP> --frames 300
```

## 3. 종합 비교 — ✅

서버 열은 **망 혼잡(실사용 공용 WiFi)** 기준 대표값. (전용 LAN best-case는 §2-(b) 참고)

| 지표 | idle (기준선) | 엣지(온디바이스) | 서버(UDP) | 서버(HTTP) |
|------|--------------|------------------|-----------|------------|
| **FPS** | — | 5.31 | **13.8** | 11.4 |
| **e2e 지연 평균** | — | 165 ms | **48.4 ms** | 62.3 ms |
| 지연 p95 | — | 242 ms | 129.3 ms | 95.3 ms |
| 추론 자체 | — | ~165 ms (Pi CPU) | **7.4 ms (GPU)** | 5.0 ms (GPU) |
| 추가 비용 | — | 없음 | 네트워크 41 ms | 네트워크 57 ms |
| **Pi CPU 사용률** | ~7~8% | **84.0%** | **17.6%** | 15.0% |
| **Pi 온도** | ~48.5 ℃ | **60.4 ℃** (max 65.2) | ~48.9 ℃² | ~49.3 ℃² |
| **Pi 온도 Δ(vs idle)** | 0 | **+11.9 ℃** | ≈0 (동률) | ≈0 (동률) |
| **Pi throttle** | `0x0` (없음) | `0x0` (스로틀 없음)¹ | `0x0` (없음) | `0x0` (없음) |
| 정확도(mAP) | — | TBD | TBD | TBD |

> **idle 기준선**: 추론/전송 아무것도 안 하는 대기 상태. CPU ~7~8% / 온도 ~48.5℃ /
> throttle 없음. 비교의 0점.
>
> ² **온도는 ambient(주변온도) 영향 ±1~2℃**: 서버 경로 온도(48.9/49.3℃)는 idle(~48.5℃)
> 대비 거의 차이 없음 — **서버 경로 ≈ idle 발열**. CPU%(idle 7% < 서버 15~17%)가 더
> 신뢰도 높은 지표이고, 둘 다 엣지(84% / 60~76℃)와는 명확히 다름.

> ¹ **엣지 throttle 주의**: 위 엣지 값은 식은 상태에서 시작한 **스로틀 없는 300프레임**
> 측정. 단, CPU가 ~84%로 계속 풀로드라 **장시간 지속 돌리면 온도가 76℃↑로 올라
> 스로틀(`0xe0000`)에 진입** → FPS 5.3 → ~4.3, 지연 165 → ~217ms로 떨어짐. 즉 표의
> 엣지 숫자는 best-case이고, 실제 지속 운용에선 발열로 성능이 깎인다.

> **Pi 자원 — 가설 입증됨**: 엣지는 Pi 4코어를 거의 풀로(CPU ~84%) 쓰며 온도가 오르는
> 반면, **서버 경로는 Pi가 추론을 안 하고 캡처+JPEG 인코딩+전송만 하므로
> CPU ~15~18% / 온도 ~49℃(idle과 동일) / 스로틀 없음**. 즉 서버 경로는 더 빠를 뿐 아니라
> Pi의 발열·전력 부담도 크게 낮음 → Pi가 다른 작업(주행·SLAM 등)에 여유를 남김.
> (출처: `server_udp.csv` / `server_http.csv`, 300프레임)

### 결론
- **서버 경로가 더 빠름** (실사용 혼잡망 기준): FPS 5.3 → 13.8 (**2.6배**), 지연 165 → 48.4ms (**3.4배↓**).
  전용 LAN(best-case)이면 27 FPS·29.7ms로 격차가 더 벌어짐(5배+).
- 이유: GPU 추론(~7ms)이 Pi CPU 추론(~165ms)보다 **약 20~30배 빠름**. 네트워크 비용
  (혼잡 41~57ms)을 더해도 여전히 엣지보다 빠르다.
- **엣지의 가치**: 네트워크/서버 없이 독립 동작, 서버 다운·네트워크 단절에 강함. 단 Pi CPU
  한계로 ~4~5 FPS + **발열/전력 제약(지속 부하 시 76℃·스로틀 → 성능 저하·하드웨어 부담)**.
- **트레이드오프**: 속도·확장성이 중요하면 서버, 자립성·오프라인이 중요하면 엣지.
- 정확도(mAP)는 라벨 테스트셋 준비되면 `eval/`로 측정해 채운다.
